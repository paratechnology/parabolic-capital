DevDash

A local developer operations dashboard for keeping multiple projects, services, file watchers, and build states visible while working across the product portfolio.

SectorInternal developer tooling
SurfaceLocal dashboard, process visibility, project tracking
StatusPrivate internal tool

DevDash is not a public product, but it matters because it shows how I improve my own delivery environment when the portfolio becomes too large to manage casually.

Private internal tool Architecture walkthrough available

What it is

DevDash is a workspace tool for seeing project status, running services, tracking local servers, and keeping development context visible across several active products.

What I built

  • Project tracking surfaces for the apps in the workspace.
  • Service visibility for local development processes and ports.
  • File and build watching concepts to reduce context switching.
  • Internal workflow design around how I actually ship the other products.

Technical surface

The technical interest is in local orchestration, process state, file-system awareness, and creating a practical operator view over many unrelated repositories.

Product judgment

I built it because the pain was real: many products, many commands, many ports, and too much memory overhead. The useful version of the tool is the one that reduces that friction without becoming another system to maintain.

I can discuss how this supports the way I work, why internal tooling matters, and where I would take it next if it became a product for teams.
